## 核心思维模型

### 1. 公开账本 -> 可控披露

以太坊默认是：

```text
规则公开
数据也大多公开
```

Midnight 的核心变化是：

```text
规则可信
但数据不必全部公开
```

项目落地：

```text
公开：
创作者收入超过某个阈值
赞助人数达到某个区间
报告周期

隐藏：
每个赞助人是谁
每笔金额是多少
完整收入流水
```

关键词：

```text
selective disclosure
选择性披露
```

### 2. 执行逻辑 -> 证明逻辑

传统智能合约更像：

```text
链上执行代码
所有节点看到状态变化
```

Midnight / ZK 更像：

```text
用户用私密数据生成证明
链上验证证明是否正确
```

项目落地：

```text
创作者不是公开所有收入明细
而是证明这些收入明细满足某个条件
```

例如：

```text
monthly_income >= 1000
supporter_count >= 50
```

### 3. 原始数据 -> 承诺 / 证明 / 结果

不要把原始敏感数据直接上链。

拆成三层：

```text
原始数据：用户自己保存
承诺/证明：证明原始数据按规则计算无误
公开结果：外界需要看到的结论
```

项目落地：

```text
原始数据：
100 笔赞助明细

证明：
这些明细加起来确实超过 1000

公开结果：
月收入 >= 1000: true
```

### 4. 用户数据 -> 用户控制

Web2：

```text
平台控制数据
```

普通公开链：

```text
所有人都能看链上行为轨迹
```

Midnight 方向：

```text
用户控制数据
```

项目落地：

```text
创作者决定：
哪些收入指标公开
哪些收入明细隐藏
哪些证明给品牌方/DAO/社区看
```

### 5. 普通 DApp -> 隐私业务场景

Midnight 不适合只做普通打赏、普通 NFT、普通转账。

它适合：

```text
需要可信
但不能全公开
```

的业务。

项目判断句：

```text
这个业务有没有“必须证明，但不能公开原始数据”的部分？
```

CreatorVault AI 的答案：

```text
有。
创作者需要证明收入能力，但不应暴露完整收入流水和赞助人关系。
```

## ZK 核心解释

ZK = Zero-Knowledge，零知识证明。

一句话：

```text
证明一件事是真的，但不暴露背后的秘密。
```

项目例子：

```text
证明创作者本月收入超过 1000 美元，
但不公开每个赞助人和每笔金额。
```

## 产品原则

必须优先体现：

- 隐私痛点真实
- 证明场景明确
- UI 简单可懂
- Demo 小而完整
- 3 分钟内能讲清楚
- README 能独立说明项目价值

避免：

- 做成普通打赏 DApp
- 做成大而全创作者平台
- 只堆 AI 功能
- 只喊隐私口号，没有 public/private/proof 区分

## 服务器和部署

使用自有服务器，不使用 Vercel / Netlify。

Demo 子域名：

```text
creatorvault.ohmycode.cc
```

服务器原则：

- 不影响 `ohmycode.cc` 现有项目。
- 不直接部署到主域名。
- 使用独立 Nginx server block。
- 使用独立目录 `/var/www/creatorvault-ai`。
- 修改 Nginx 前先备份。
- reload 前必须 `nginx -t`。

不要把服务器密码、私钥、API key 写进仓库。
